# AlphaMining 日频复现运行手册

## 1. 适用范围

本手册用于从零运行日频版 GFlowNet + AlphaEval + LightGBM + RQAlphaPlus 流水线。行情数据只从本地 `data/price.csv` 读取，不调用外部数据接口。

完整流程如下：

```text
price.csv
  ↓
数据检查、复权、异常处理与截面标准化
  ↓
表达式引擎与 GFlowNet 因子挖掘
  ↓
AlphaEval 评价与 DPP 去相关筛选
  ↓
LightGBM 滚动融合，预测 t+1 到 t+5 收益
  ↓
RQAlphaPlus Top N 等权策略回测
```

## 2. 运行环境分工

| 阶段 | 推荐环境 | 必要条件 |
|---|---|---|
| 数据准备、表达式、GFlowNet、AlphaEval、LightGBM | Google Colab | NVIDIA A100、CUDA、PyTorch |
| 单元测试与结果检查 | 本地电脑或 Colab | Python 3.10 及以上 |
| 策略回测 | 本地已获授权的 RQAlphaPlus 环境 | Colab 产物、`rqalpha_plus` 与本地数据包 |

RQAlphaPlus 不在 `requirements.txt` 中，因为它是授权软件，必须通过米筐授权渠道单独安装。

### 5.2 字段要求

| 规范字段 | 必需 | 常见可识别别名示例 | 说明 |
|---|---:|---|---|
| `date` | 是 | `trade_date`、`datetime`、`日期`、`交易日期` | 交易日期 |
| `code` | 是 | `symbol`、`ticker`、`证券代码`、`股票代码` | 股票标识 |
| `open` | 是 | `open_price`、`开盘价` | 开盘价 |
| `high` | 是 | `high_price`、`最高价` | 最高价 |
| `low` | 是 | `low_price`、`最低价` | 最低价 |
| `close` | 是 | `close_price`、`收盘价` | 收盘价 |
| `volume` | 是 | `vol`、`trade_volume`、`成交量` | 成交量 |
| `amount` | 否 | `turnover`、`trade_amount`、`成交额` | 成交额 |
| `vwap` | 否 | `avg_price`、`average_price`、`成交均价` | VWAP；缺失时自动估算 |
| `adj_factor` | 否 | `adjust_factor`、`factor`、`复权因子` | 时点可得的复权因子 |
| `industry` | 否 | `industry_code`、`申万行业`、`行业` | 行业风险暴露 |
| `market_cap` | 否 | `total_market_cap`、`float_market_cap`、`市值` | 市值风险暴露 |

最小示例：

```csv
date,code,open,high,low,close,volume,amount
2020-01-02,000001.SZ,16.65,16.95,16.55,16.87,153023500,2578400000
2020-01-03,000001.SZ,16.94,17.31,16.92,17.18,111619400,1911550000
```

### 5.3 数据边界

- 每一行应代表一只股票的一个交易日；
- 同一 `date + code` 若重复，保留最后一行；
- 价格必须为正，且 `low <= high`；
- 缺失价格只在同一股票内部前向填充；
- `adj_factor` 必须是当时已经可得的数据，不能使用事后修订信息；
- 行业和市值字段缺失时可以运行，但对应风险惩罚不会启用；
- LightGBM 默认至少需要约 504 个训练交易日，建议准备明显更长的历史区间。

### 8.1 快速生成首个模型


这是用于尽快验证端到端可运行并产出模型的配置，不用于最终研究结论。快速流程成功后，将 `FAST_MODE=False` 切换到 `configs/training_config.yaml` 进行正式全量实验。原始字段 `total_turnover` 会自动映射为 `amount`，并用于计算真实 `vwap=amount/volume`。

```bash

```

训练时每条轨迹都会输出：

- 当前 epoch、当前 step、全局 step 与总体完成百分比；
- 生成的完整表达式及动作数量；
- reward、RankIC、LongIR、风险惩罚、观测覆盖率、有效交易日覆盖率与覆盖率惩罚；
- `logPF` 和该轨迹的 Trajectory Balance loss；
- 单轨迹耗时与训练累计耗时。

每个 epoch 完成反向传播和参数更新后还会打印：

- 当前 epoch 与总 epoch；
- Trajectory Balance loss；
- 平均奖励、最高奖励、平均 RankIC 和平均覆盖率；
- `logZ`、梯度范数和学习率；
- 单轮耗时与累计耗时；
- 批量轨迹采样耗时与 Reward 计算耗时；
- 最佳 checkpoint 是否保存；
- 子表达式缓存的命中数、未命中数、并发等待数、命中率、节点数和内存占用；
- PyTorch/Pandas 时序算子调用次数和 CUDA 算子累计耗时；
- A100 已分配显存、保留显存和单轮峰值显存。

逐轨迹明细保存在 `results/gflownet_trajectory_metrics.csv`，逐 epoch 汇总保存在 `results/gflownet_training_metrics.csv`，便于训练后绘图分析。训练完成后的因子池生成阶段也会为每次采样输出 attempt、接受、重复或低覆盖拒绝状态、当前唯一表达式数量、reward 和表达式。

覆盖率只在未来收益标签有效、且当日股票数达到 `min_cross_section` 的样本上计算。系统同时统计因子有效观测占比和有效交易日占比，并取两者较低值：

```text
CoveragePenalty = min(1, (effective_coverage / min_coverage) ^ coverage_penalty_power)
最终 Reward = 原始 Reward × CoveragePenalty
```

默认 `min_coverage: 0.80`、`coverage_penalty_power: 2.0`。低于 80% 的表达式在训练中会被降权，并在生成及保存因子池时被硬过滤，不会进入 AlphaEval、LightGBM 或本地 RQAlphaPlus 回测。`alpha_pool.csv` 会额外记录 `coverage`、`valid_date_coverage` 和 `coverage_penalty`，排查稀疏因子时可直接查看。

加速实现默认开启：同一 epoch 的全部轨迹使用批量 Transformer 推理；批内重复表达式只评价一次；Reward 使用 `training.reward_workers` 个线程并行；RankIC、Top 10% LongIR、行业暴露和市值暴露使用向量化计算；A100 启用混合精度与 TF32。

九个时序算子 `ts_mean`、`ts_std`、`ts_rank`、`ts_delay`、`ts_delta`、`ts_sum`、`ts_max`、`ts_min`、`ts_zscore` 默认使用 `auto` 后端：Colab A100 上解析为 `torch/cuda`，无 CUDA 的本地环境自动回退到 Pandas。PyTorch 后端先把有序行情组织成“股票 × 时间”张量，再按股票分块滚动，保留原实现的窗口预热、缺失值、`std(ddof=1)` 和 `ts_rank` 并列排名语义。训练开始日志必须出现 `ts_backend=torch ts_device=cuda`，否则不要把该次运行视为 A100 GPU 正式训练。

```yaml
operators:
  time_series_backend: auto
  time_series_device: auto
  time_series_chunk_size: 64
  time_series_dtype: float32
```

`time_series_chunk_size` 是每次送入滚动窗口的股票数，不是交易日数。A100 40/80GB 通常可从 64 开始；如果显存充足可尝试 128，如果遇到 CUDA OOM 则降至 32 或 16。正式训练推荐 `float32`；`float64` 更接近 Pandas 数值但在 A100 上明显更慢。需要强制校验 CUDA 时可以把后端和设备分别改为 `torch`、`cuda`，此时 CUDA 不可用会直接报错。

表达式执行器会将每个结构化节点作为缓存键，因此 `cs_rank(ts_mean(close,20))` 与 `add(ts_mean(close,20),volume)` 会共享一次 `ts_mean(close,20)` 计算。多个 Reward 线程同时请求同一节点时使用 single-flight：第一个线程负责计算，其余线程等待同一个结果，不会重复占用 CPU。LRU 同时受节点数和内存限制，默认最多 128 个节点、512 MB；达到任一上限会优先淘汰最久未使用的节点。完整因子池重算也复用同一机制。

快速配置可调整：

```yaml
reward:
  subexpression_cache_enabled: true
  subexpression_cache_max_entries: 128
  subexpression_cache_max_mb: 512
```

日志中的 `cache_hit_rate` 是累计命中率，epoch 指标文件还包含当轮 `cache_hits`、`cache_misses`、`cache_waits`、`cache_evictions`、缓存节点数和 RAM 占用。开始阶段命中率偏低属正常现象；随着共享子树增多通常会逐步提高。如果 Colab RAM 紧张，先把 `subexpression_cache_max_mb` 降到 256；如果内存充足但淘汰频繁，可提高到 1024。默认 `reward_workers: 4`，如果 Colab CPU 利用率仍低可尝试 6 或 8，若内存压力明显则降为 2。

正式训练会保存并重新加载检查点。成功标志：

```text

results/gflownet_training_metrics.csv
results/gflownet_trajectory_metrics.csv
results/alpha_pool.csv
results/alpha_factor_matrix.pkl
results/alpha_factor_matrix_oos.pkl
```

`alpha_pool.csv` 记录 `factor_001`、`factor_002` 等因子名称、表达式和奖励信息；`alpha_factor_matrix.pkl` 保存包含训练历史的完整因子值，供 LightGBM 滚动训练；`alpha_factor_matrix_oos.pkl` 只保存 2023 年以后的样本外因子值，便于检查和归档。RQAlphaPlus 仍只读取 2023 年以后的 `prediction_score.csv`，不会直接读取样本内标签。

## 9. 阶段四：AlphaEval 筛选


## 12. 阶段六：本地 RQAlphaPlus 回测

### 12.1 前提

本阶段不在 Colab 运行。在本地已获授权的环境中安装 `rqalpha_plus`，准备好本地数据包，并把 Colab 下载的 `alphamining_colab_outputs.zip` 放到仓库根目录。本项目不会自动下载 RQAlphaPlus 数据包。

### 12.2 导入 Colab 产物

推荐打开并运行：

```text
notebooks/06_rqalpha_backtest.ipynb
```

Notebook 会把压缩包解压到原有目录结构，并校验 `results/lightgbm/prediction_score.csv`。压缩包同时保留原始因子池、因子矩阵和模型，便于本地审计；RQAlphaPlus 策略直接读取的是因子经 LightGBM 融合后的预测分数。

### 12.3 执行

```bash
python -m rqalpha_strategy.run_backtest \
  --bundle ~/.rqalpha-plus/bundle \
  --predictions results/lightgbm/prediction_score.csv \
  --output-dir results/backtest_report \
  --initial-cash 1000000 \
  --benchmark 000300.XSHG \
  --top-n 30 \
  --rebalance-days 5 \
  --slippage 0.001
```

### 12.4 防止未来数据泄漏

策略只选择 `signal_date < trade_date` 的最近信号。同日收盘数据生成的分数不会在同一根日线成交。RQAlphaPlus 负责 T+1、整手约束、涨跌停、成交量限制、佣金、印花税、滑点及成交记录。

### 12.5 回测输出

成功后，`results/backtest_report/` 至少应包含：

```text
backtest_result.pkl
backtest_summary.json
equity_curve.png
```

RQAlphaPlus 分析器还会生成可用的绩效报告、持仓和交易明细。重点核对年度收益、总收益、Sharpe、最大回撤、波动率、换手率、净值曲线与交易历史。

## 15. 常见问题

### 15.1 提示没有找到 `data/price.csv`

本地运行时确认文件路径严格为 `data/price.csv`。Colab 运行时确认 Google Drive 中存在 `MyDrive/price.csv`，并重新执行 Drive 挂载与复制单元格。

### 15.2 提示缺少必需字段

打开 `price.csv` 第一行，检查日期、代码、开高低收和成交量字段是否存在。若字段名不在自动别名表中，可先改为规范名称；不要把索引列误当作股票代码。

### 15.3 CUDA 不可用或 GPU 不是 A100

在 Colab 重新选择 A100 运行时，然后从 GPU 检查单元格重新执行。若无法分配 A100，请不要用 `--allow-non-a100` 生成正式结果。

### 15.4 macOS 导入 LightGBM 失败

先执行 `brew install libomp`，重启终端并重新激活虚拟环境。仍失败时重新安装 LightGBM wheel。

### 15.5 AlphaEval 或 LightGBM 可用样本过少

确认股票数量足够、历史长度明显超过 504 个交易日，并检查大窗口因子是否产生过多空值。行业、市值字段缺失不会阻止运行。

### 15.6 RQAlphaPlus 无法导入

确认当前运行的是本地 Notebook，而不是 Colab。这不是公开 `pip` 依赖缺失；请切换到已通过米筐授权渠道安装 `rqalpha_plus` 的本地 Python 环境。

### 15.7 没有找到 Colab 训练产物

确认浏览器已完成下载，并把 `alphamining_colab_outputs.zip` 放到本地仓库根目录。不要更改压缩包内部的 `results/` 和 `checkpoints/` 目录结构。

### 15.8 回测没有交易

检查预测文件是否包含 `signal_date`、`code`、`prediction_score`，日期是否覆盖 RQAlphaPlus 数据包，股票代码能否映射为 `.XSHG`、`.XSHE` 或 `.XBSE`，以及第一天没有更早信号是否属于预期。
